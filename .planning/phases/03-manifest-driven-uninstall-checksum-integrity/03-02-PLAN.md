<objective>
Deliver the one genuinely-new capability this milestone's trust goal still needs: published, verifiable SHA256 checksums for every release artifact (INTEG-02). Per D-05/D-06/D-07, create `release/checksums.js` (matching the zero-dependency, host-agnostic style of its `release/version.js` / `release/github-release.js` siblings), generating a single `releases/SHA256SUMS.txt` (standard `shasum -a 256 -c` two-space text convention — not per-file sidecars), wired to run automatically during `npm run package`/`publish:*` and uploaded to the GitHub Release alongside the `.ccx`.

Per D-08, write a new `VERIFY.md` with plain-language, copy-paste verification steps for both macOS (`shasum -a 256 -c SHA256SUMS.txt`) and Windows (`certutil -hashfile` primary, `Get-FileHash` alternative), linked from the README.

Checksum generation uses Node's built-in `crypto` module streamed over `fs.createReadStream()` — no new npm dependency, platform-agnostic. VERIFY.md must frame checksums honestly as integrity-against-corruption/simple-tampering, NOT authenticity (no signing key — signing is deferred to v2 SECV2-01).

Purpose: Let a zero-terminal-experience user independently confirm a download matches what was published.
Output: `release/checksums.js`, `src/__tests__/checksums.test.ts`, `VERIFY.md`, wired `package.js` + `github-release.js`, README link.
</objective>

| T-3-V6 | Cryptography | release/checksums.js (SHA256 integrity) | medium | mitigate | Use Node built-in `crypto.createHash('sha256')` streamed over `fs.createReadStream()`; never hand-roll a hash (Don't Hand-Roll) |
| T-3-04 | Tampering | SHA256SUMS.txt malformed (single-space / CRLF) silently fails `shasum -c` | medium | mitigate | `formatChecksumLine` emits exactly two-space separator + bare filename + LF; `checksums.test.ts` asserts the format (Pitfall 1) |
