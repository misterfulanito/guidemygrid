<objective>
Build the corrected `.ccx` packaging pipeline and prove, end-to-end on the real dev Mac, that it actually installs — the thing the original Plans 03-04 never achieved because they built on a disproven assumption (see `01-RESEARCH.md`'s CRITICAL ADDENDUM). Plan 03 already fixed `manifest.json` (object-shaped `host`, portal-issued `id`) and removed every file implementing the old raw-copy/native-dialog installer. This plan:

1. Creates `distribution/photoshop/build-ccx.js`: a Node script that zips `dist/` into `releases/GuideMyGrid-v<version>.ccx`, matching the exact structure confirmed by inspecting a real shipping competitor's `.ccx` (a top-level `dist/` folder inside the zip, not files at the zip root). This lives directly under `distribution/photoshop/`, not inside `macos/`, because `.ccx` packaging is identical on macOS and Windows — it is not an OS-specific mechanism, unlike the retired `.app`/`.dmg` build was.
2. Fixes `scripts/package.js`'s existing `.ccx`-creation step, which currently zips `dist/`'s *contents* directly at the zip root (the same wrong structure this plan's research disproved) — it now delegates to the new script instead.
3. Wires the `.ccx` into `release/github-release.js`'s publish flow, removing the now-dead `.dmg` reference (that file already referenced `.ccx` from this project's pre-`.pkg` history — it just also still carried the `.dmg` line added by the retired approach).
4. Ends with a blocking manual QA checkpoint: build the `.ccx`, double-click it, confirm Creative Cloud Desktop launches and shows the "unverified third-party developer" warning (not a Marketplace-review pass), confirm zero admin/root prompt, confirm the plugin appears in Photoshop's Plugins menu, confirm the panel opens and works.

SCOPE NOTE (surfaced, not silently decided): `scripts/package.js`'s step 2 (`-installer.zip`, bundling raw `dist/` + the Windows install/uninstall scripts, marketed as "no Creative Cloud needed") rests on the exact same raw-file-copy assumption this milestone's research just disproved for macOS — it is very likely equally non-functional. Fixing or removing it is out of this plan's scope: Phase 1 is macOS-only per ROADMAP.md, and REQUIREMENTS.md already flags that Phase 2 must re-verify the Windows install model before building on it. This plan does not touch that step.

If Task 2's manual QA reveals the `.ccx` failing to install specifically because Creative Cloud Desktop rejects `manifestVersion: 4` (not because of the `host`/`id` fix — that's already applied), that is new evidence for a possible v5 escalation, per D-01a's own fallback discipline — do not silently bump the manifest version to fix it; surface the finding instead.

Purpose: Ship the real, working macOS install path — the one no prior plan in this phase actually achieved.
Output: `distribution/photoshop/build-ccx.js`, a built `.ccx`, and a verified Creative Cloud Desktop install flow.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `build-ccx.js` (build machine) → `releases/*.ccx` (distributed artifact) | The zip's contents become exactly what Creative Cloud Desktop installs on every user's machine — anything staged incorrectly here ships to every install |
| Double-clicked `.ccx` (user machine) → Creative Cloud Desktop | CC Desktop, not our code, evaluates trust and performs the actual install steps; our code's only leverage is what's inside the zip |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-01-17 | Tampering | Incorrect zip structure could ship a `.ccx` that silently fails to install (a "corrupt what ships" surface, listed under Tampering since the artifact itself is what's malformed) | medium | mitigate | `build-ccx.js` stages `dist/` under a `dist/` folder name before zipping, matching the confirmed real-world structure exactly; Task 1's automated verify inspects the produced zip's actual file listing, not just that a file was created |
| T-01-18 | Elevation of Privilege | Any accidental reintroduction of a privileged install step into the new packaging path | low | mitigate | `build-ccx.js` only ever invokes the unprivileged `zip` utility as the invoking user, matching the existing `scripts/package.js` convention; the actual "install" step is entirely Creative Cloud Desktop's, outside this plan's code |
| T-01-19 | Tampering | Supply chain — this plan adds no new package | high | mitigate | No new npm package introduced; `create-dmg` was already removed in Plan 03; the `zip` utility used is the same OS-provided binary `scripts/package.js` already relied on for the `-installer.zip` step |
</threat_model>
