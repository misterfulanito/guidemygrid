<objective>
Phase 1's macOS installer was built and shipped (Plans 03–04, original execution) on a wrong technical assumption: that copying the plugin's `dist/` output directly into Photoshop's `PluginsStorage/PHSP/<ver>/Plugin/<id>/` folder would make Photoshop list it. Manual QA on the real dev Mac (documented in `01-RESEARCH.md`'s "CRITICAL ADDENDUM" and its follow-up, both right after `## Summary`) proved this false: Photoshop's Plugins panel only shows what Creative Cloud Desktop's own install agent (UPIA) has registered, and UPIA only registers what it installed itself — a Marketplace submission or a `.ccx` file, never a raw filesystem copy. The only viable no-Marketplace path is a `.ccx` (a plain zip of `dist/`) that the user double-clicks, which launches Creative Cloud Desktop and installs the plugin at user level with an "unverified third-party developer" warning.

This plan does the groundwork the `.ccx` approach requires and retires everything built on the disproven assumption:
1. Get a portal-issued plugin `id` (a one-time, free, human-only step — Adobe's own tooling auto-generates this when you create a Draft listing, and creating a Draft never triggers a review).
2. Fix `manifest.json`'s `"host"` field from an array to a bare object (confirmed via direct inspection of a real shipping competitor's `.ccx`) and apply the new `id`. `manifestVersion` stays `4` per the already-locked D-01a decision — do NOT bump to 5 here; only escalate if Plan 04's install QA surfaces real evidence v4 fails.
3. Delete every file that implemented the disproven raw-copy/native-dialog installer (`install-payload.sh`, `installer.applescript`, `build-installer.js`, and their two Jest tests), plus the `build:mac-installer` npm script and the `create-dmg` devDependency those files needed.
4. Correct the docs/scope-notes that still describe the retired approach as current.

SCOPE NOTE (surfaced, not silently decided): CONTEXT.md's D-02 (confirm dialog), D-03 (app labeled "Install GuideMyGrid"), D-04 (Photoshop-running hard block), and D-05 (success dialog) described a custom installer `.app`'s own native dialogs and its own process check. None of this is buildable anymore — Creative Cloud Desktop owns 100% of a `.ccx`'s install UX (its own warning dialog, its own install sequence); our code has no hook into any of it. This directly parallels REQUIREMENTS.md's already-recorded supersession of MAC-02/MAC-03. Task 3 appends a short supersession note to CONTEXT.md so this isn't silently dropped. D-01, D-01a, and D-06 are unaffected.

Also flagged, not fixed here: `01-PATTERNS.md`'s `build-installer.js`/`installer.applescript` pattern write-ups and its two test-file sections now describe deleted code — treat that file's AppleScript-specific guidance as void for this rework; nothing in this plan depends on it, and no task edits it.

Purpose: Clear the ground truth (manifest schema + plugin identity) and remove the disproven code so Plan 04 can build a `.ccx` that actually installs.
Output: A corrected, portal-registered `manifest.json`; a `distribution/photoshop/macos/` directory with no installer logic left in it; accurate docs.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| manifest.json → Creative Cloud Desktop's install agent (UPIA) | UPIA parses this file to decide how/whether to install the plugin; a malformed `id`/`host` causes a silent install failure, not a security bypass |
| Adobe Developer Distribution portal → repo | The portal-issued plugin id is copied by hand into manifest.json; a wrong/mistyped id would misidentify the plugin, not grant excess privilege |
