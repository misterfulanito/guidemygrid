<objective>
Correct the two documentation files still describing Windows as architecturally different from macOS, and add the rescoped WIN-05 CI verification job. Phase 2's central finding (D-01) is that Windows uses the identical `.ccx` + Creative Cloud Desktop mechanism as macOS — but `distribution/README.md` still claims "Windows has no equivalent Creative Cloud Desktop dependency" and `distribution/photoshop/windows/README.md` still says the real Windows rework is pending in a future phase. Both are now false and must match the confirmed parity reality.

WIN-05 as literally worded ("CI runs the installer and uninstaller end-to-end, confirms no UAC prompt, correct install path, clean uninstall") is not achievable: Creative Cloud Desktop requires an interactive GUI and an authenticated Adobe login and cannot be driven headlessly in CI. That real-device verification is explicitly deferred (D-06) to before ship, not before this phase. This plan builds what CI *can* actually do — a `windows-latest` job that builds and packages the `.ccx` on a real Windows runner (proving the OS-agnostic pipeline works on Windows), then extracts and inspects the artifact to assert the two regression properties that matter: no `requiredPermissions` in the manifest (the elevation-prompt trigger) and none of the five retired raw-copy scripts present inside the shipped `.ccx`.

This plan depends on Plan 01: the scripts are deleted and `scripts/package.js` is fixed there, so the docs here can accurately describe a repo where the retirement has already landed, and the CI job runs against a cleaned tree.

Purpose: Make the distribution docs tell the truth about Windows parity, and add the first automated CI guard for the release artifact.
Output: Two corrected README files and a new `windows-latest` GitHub Actions workflow.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| built .ccx artifact → CI inspection | The CI job treats the freshly built `.ccx` as untrusted-until-verified: it parses dist/manifest.json (safe ConvertFrom-Json, no external input beyond the repo's own build output) and scans extracted contents. |
| repo → GitHub Actions runner permissions | The verification job runs with no write permissions to the repo or releases; it only reads, builds, and asserts. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-04 | Tampering | manifest.json regaining a `requiredPermissions` block in a future commit, silently reintroducing CC Desktop's admin-password prompt on Windows installs | high | mitigate | The pwsh step throws if `requiredPermissions` is present in the built artifact's manifest — CI-side twin of Plan 01's Jest guard, run on every push/PR |
| T-02-05 | Tampering / Repudiation | A retired raw-copy script reappearing inside the shipped `.ccx` (not just the source tree) | medium | mitigate | The pwsh step scans the extracted `.ccx` and throws if any of the five retired filenames is found |
| T-02-06 | Elevation of Privilege | The CI workflow itself carrying broader repo permissions than it needs | low | mitigate | No top-level `permissions:` block is declared (release.yml's `contents: write` is deliberately omitted) — least privilege for a read-and-assert job |
| T-02-SC | Tampering | Supply-chain risk from package installs | low | accept | No new npm/pip/cargo packages are added (RESEARCH confirmed zero); `npm ci` installs the existing pinned lockfile only — no legitimacy checkpoint required |
